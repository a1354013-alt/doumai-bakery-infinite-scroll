# 單元測試與邊界條件：面試加分指南

在面試中，展示這份測試案例不僅是展示「我會寫測試」，更是展示你的**工程思維**。以下是你可以運用的面試說詞與優勢分析：

---

## 1. 為什麼要處理邊界條件 (Edge Cases)？

**面試說詞建議：**
> 「在開發購物車功能時，我特別針對邊界條件編寫了單元測試。例如：當使用者嘗試將數量設為負數、或超過庫存上限時，系統應該如何反應。這能確保業務邏輯的健壯性，避免前端出現非預期的錯誤狀態。」

**展現的特質：**
- **嚴謹性**：你考慮到了使用者可能產生的異常操作。
- **防禦性編程 (Defensive Programming)**：你在邏輯層就過濾了錯誤資料，而非依賴 UI 限制。

---

## 2. 為什麼要將邏輯抽離成純函數 (Pure Functions)？

**面試說詞建議：**
> 「我將購物車的計算與增刪邏輯抽離到獨立的 `utils/cartLogic.js` 中，並確保它們都是純函數。這樣做的好處是邏輯與 Vue 組件解耦，不僅讓測試變得非常容易，未來如果要遷移框架或引入 Pinia，這些核心邏輯都可以直接複用。」

**展現的特質：**
- **架構設計能力**：懂得關注點分離 (Separation of Concerns)。
- **可維護性**：程式碼結構清晰，易於擴展。

---

## 3. 測試案例的具體價值

| 測試場景 | 展現的專業點 |
| :--- | :--- |
| **空值與異常資料處理** | 展現對資料安全與程式碼健壯性 (Robustness) 的重視。 |
| **庫存上限攔截** | 展現對業務規則 (Business Rules) 的精準實作。 |
| **自動修正非法輸入** | 展現對使用者體驗 (UX) 的細膩考量，不只是報錯，而是引導至正確狀態。 |

---

## 4. 如何在面試中展示？

當面試官問到：「你如何確保程式碼品質？」或「你遇過最難處理的 Bug 是什麼？」時，你可以順勢提到：
> 「我習慣為核心邏輯編寫單元測試。以這個專案為例，我使用 Vitest 涵蓋了購物車的各種邊界場景。這讓我在重構 UI 或增加新功能時，能有十足的信心確保核心業務邏輯不會壞掉。」

**這句話能立刻將你與一般的初階開發者區分開來。**
